# v1.2.0 Roadmap

This roadmap defines the recommended scope for the next release after `v1.1.0`.

The goal of `v1.2.0` should be to deepen the fluent API in the places developers use every day, without overreaching into broad PHPUnit parity claims that the project is not ready to maintain yet.

## Release Goal

`v1.2.0` should make `php-testify` better at:

- reusable parameterized tests
- clearer test selection and filtering
- richer assertion coverage
- better fluent metadata controls
- more explicit, reliable developer workflows

This should be a focused feature release, not a general rewrite.

## Recommended Theme

Primary theme:

- make the fluent API easier to reuse and scale in real projects

That means prioritizing:

1. reusable datasets
2. better CLI selection ergonomics
3. richer `expect()` matchers
4. conditional execution controls
5. documentation and reporting quality

## Milestone 1: Reusable Datasets

Goal:

- move from inline-only datasets to reusable datasets that can be shared across specs

Scope:

- named dataset registry
- dataset lookup from fluent tests
- named dataset output labels

Task breakdown:

- `[ ]` Design a global dataset registration API
- possible direction: `dataset('users', [...])`
- `[ ]` Support `it(...)->with('users')`
- `[ ]` Keep support for inline `->with([...])`
- `[ ]` Support named datasets with associative labels
- `[ ]` Decide merge behavior when combining inline and registered datasets
- `[ ]` Add validation for unknown dataset names
- `[ ]` Show dataset labels clearly in CLI output
- `[ ]` Add regression tests for dataset registration and reuse

Exit criteria:

- developers can define reusable datasets once and consume them across multiple specs

## Milestone 2: Better Test Selection And Discovery

Goal:

- make the CLI practical for larger projects where substring filtering becomes too loose

Scope:

- exact-path filtering
- listing discovered tests
- clearer selection semantics

Task breakdown:

- `[ ]` Add exact suite/test path filtering
- possible direction: `outer > inner::works`
- `[ ]` Keep substring `--filter` for broad matching
- `[ ]` Add a `--list` mode for discovered fluent tests
- `[ ]` Include dataset-expanded case names in list output
- `[ ]` Improve verbose output so suite path and dataset labels are easier to scan
- `[ ]` Define how class-based PHPUnit tests and fluent specs should appear in listing mode
- `[ ]` Add regression tests for selection behavior

Exit criteria:

- developers can confidently target one suite, one test, or one dataset case without trial-and-error filtering

## Milestone 3: Assertion Expansion

Goal:

- make `expect()` cover more real testing workflows without forcing users back to PHPUnit assertions

Scope:

- structured data assertions
- output assertions
- file assertions
- better failure messages

Task breakdown:

- `[ ]` Add JSON assertions
- possible directions: `toMatchJson()`, `toHaveJsonPath()`, `toDecodeTo()`
- `[ ]` Add output assertions for callables
- possible directions: `toOutput()`, `toOutputContains()`
- `[ ]` Add file-system assertions
- possible directions: `toExist()`, `toBeFile()`, `toBeDirectory()`, `toHaveContents()`
- `[ ]` Add stronger array assertions
- possible directions: subset checks, ordered checks, contains-associative checks
- `[ ]` Improve failure messaging for arrays and strings
- `[ ]` Decide where lightweight diffs are worth adding
- `[ ]` Add regression tests for each matcher family

Exit criteria:

- common application-level tests can stay inside the fluent API more often

## Milestone 4: Conditional Execution Controls

Goal:

- improve practical test-state controls beyond static skip/incomplete markers

Scope:

- conditional skip helpers
- environment guards
- suite-level controls

Task breakdown:

- `[ ]` Add `skipIf()` and `skipUnless()` helpers
- `[ ]` Add PHP-version guards
- possible direction: `requiresPhp('>=8.2')`
- `[ ]` Add extension guards
- possible direction: `requiresExtension('intl')`
- `[ ]` Add OS guards
- possible direction: `requiresOs('Windows')`
- `[ ]` Decide whether guards should mark tests skipped or incomplete
- `[ ]` Add suite-level skip support
- `[ ]` Add regression tests for each guard path

Exit criteria:

- developers can express common environment-sensitive test conditions fluently and predictably

## Milestone 5: Reporting And Documentation

Goal:

- make the project easier to adopt by being more explicit and easier to debug

Scope:

- CLI clarity
- supported-vs-missing documentation
- release discipline

Task breakdown:

- `[ ]` Improve dataset and group visibility in verbose output
- `[ ]` Consider simple machine-readable output if CI consumers need it
- `[ ]` Add README examples for reusable datasets and conditional skips
- `[ ]` Update the compatibility checklist after feature implementation
- `[ ]` Add a release-prep checklist for `v1.2.0`
- `[ ]` Keep unsupported PHPUnit parity cases clearly documented

Exit criteria:

- users can understand what the library supports without reading source code

## Explicit Non-Goals For v1.2.0

These items are valid future work, but should not be allowed to bloat this release unless there is a strong reason:

- full PHPUnit attribute parity
- dependency chaining equivalent to `#[Depends]`
- XML configuration bridging
- event/listener/extension compatibility
- full mock/stub abstraction layer
- coverage integration redesign

## Release Gate

Before tagging `v1.2.0`, require all of the following:

- `[ ]` local `composer ci` passes
- `[ ]` GitHub Actions matrix is green
- `[ ]` each new fluent API has regression coverage
- `[ ]` README examples match shipped behavior
- `[ ]` changelog entry is finalized before tagging
- `[ ]` compatibility checklist is updated

## Suggested Versioning Strategy

- `v1.1.x` for bug fixes and stability improvements to the new fluent metadata system
- `v1.2.0` for the next planned feature release described here
- avoid `v2.0` unless execution semantics or public fluent APIs intentionally break
